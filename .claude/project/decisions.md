# Decision Log (ADRs)

Append-only record of significant decisions. Newest first. Format: date · decision · why.

## 2026-07-18 · Candidate profile & seniority fit (F3)
Added `services/ai/profile_service.py` — deterministic extraction of name/title/total-years from the parsed resume text + seniority-band classification, compared against the JD's stated "N+ years" to yield Below/Meets/Exceeds/Unknown. Agent 1 computes it and attaches `candidate_profile` to the evidence; evaluation carries it onto `HiringReport.candidate_profile` (both paths). Optional/nullable for backward-compat. **Why:** pure skill-fit ignores "who is this and are they senior enough" — a core recruiter question. Kept deterministic (regex/heuristics, no hardcoded names) on the evidence side; the LLM never sees the raw resume (Golden Rule 2). 120 tests pass. Frontend handoff + example payload updated. Held next: chunking, hybrid, multi-candidate ranking.

## 2026-07-18 · Provider-agnostic LLM engine + grounded reasoning core (F1)
Reasoning must never be tied to a vendor/model. Added `services/ai/llm_service.py` — one factory driven by `LLM_PROVIDER` + `LLM_MODEL` (env), supporting openai / anthropic / google-gemini (aliases: gemini→google, claude→anthropic), lazy per-provider imports, per-provider default models, returns None → deterministic fallback. Installed `llama-index-llms-google-genai`; config reads `GOOGLE_API_KEY` or `GEMINI_API_KEY`. Removed the hardcoded provider imports + stale `claude-3-5-sonnet-20240620` id from `evaluation_service`. Rebuilt the LLM path into testable helpers: `_build_reasoning_prompt` (grounded, per-requirement evidence + must/nice importance, evidence-only), `_extract_json` (fence/prose-tolerant), `_finalize_llm_report` (deterministic ownership: quality+authenticity+weighted coverage+importance computed in code, overall recombined). Tested with a fake LLM (no network). **Why:** user directive — avoid vendor lock-in, current key is Gemini Pro; and the mock was the only working "reasoning". Golden Rule 2 preserved (LLM sees only retrieved evidence). Default model chosen fully config-driven (LLM_MODEL). F3 (candidate profile/seniority) deferred. 115 tests pass.

## 2026-07-18 · Requirement prioritization (must-have vs nice-to-have) + weighted coverage (F2)
Every requirement was scored with equal weight, so missing a nice-to-have hurt as much as missing a must-have. Added deterministic `classify_priorities(jd_text, requirements)` in the extractor: JD priority headers ("Nice to have:", "Requirements:") set a running context, inline cues ("a plus", "required") override it, default is must-have. Agent 1 attaches `importance`+`weight` to each evidence item; `coverage_score` is now importance-weighted (`_weighted_coverage`) in both mock and LLM paths; `RequirementFit` gained optional `importance`/`weight`; missing must-haves are surfaced first in recruiter weaknesses. Defaults (weight 1.0) keep all prior behavior/tests identical. **Why:** recruiter relevance + it is the enabler F1 (grounded reasoning) and future candidate ranking both depend on. Chosen order was F2→F1. Deterministic baseline now; F1 adds an LLM refinement pass over it. 107 tests pass.

## 2026-07-18 · Generalized (deterministic) JD requirement extraction + generic tech-specificity
`jd_requirement_extractor` was pure keyword lookup against `TECH_TAXONOMY` — any skill off that ~100-term list was invisible, so evaluation was silently capped to known tech. Rebuilt as two deterministic layers: (1) taxonomy precision match, (2) grammar-cue extraction that pulls unknown tools, soft skills, domain phrases, and seniority ("5+ years") straight from the JD text (bullets / cue lines / requirement blocks), with bounded, deduped, noise-filtered phrases. Also replaced the second hardcoded keyword regex in `retrieval_service` (tech-specificity signal) with a generic morphology+taxonomy signal. **Why:** the vision explicitly rejects keyword matching and Golden Rules 4–5 demand algorithms-over-examples that generalize to any JD. Kept extraction deterministic (not LLM) because the JD is a raw document and Golden Rule 2 bars the LLM from raw docs. Verified: retrieval math is genuine cosine (BGE normalized + FAISS IndexFlatIP — probe-confirmed, no bug). 101 tests pass.

## 2026-07-18 · Deterministic authenticity / keyword-stuffing detection + real quality_score
New `services/ai/authenticity_service.py` computes credibility from evidence sections: a skill is *demonstrated* only if it has Experience/Projects evidence; skills appearing solely in Skills/Summary are *over-claimed* (keyword stuffing). Emits `AuthenticityAssessment` (credibility_score, keyword_stuffing_risk, over_claimed_skills, corroboration_ratio, explanation) on `HiringReport` and replaces the hardcoded `quality_score = 85` with a corroboration+depth blend. Computed on **both** the LLM and mock paths — never delegated to the LLM. Field is `Optional` for backward-compat with pre-existing stored reports. **Why:** the vision explicitly demands "is the candidate exaggerating / keyword stuffing?"; the hardcoded 85 violated algorithms-over-examples. Fits the two-agent model as a deterministic service Agent 2 consumes evidence-first — no new agent, fully explainable.

## 2026-07-17 · Parallel dev via git worktrees (2 + main hub)
Two worktrees (`cc2-backend` → `dev/backend`, `cc2-frontend` → `dev/frontend`) plus the main worktree as integration/QA/push hub. QA is on-demand in the main worktree, not a 3rd permanent worktree. **Why:** the main worktree already has the full env; a dedicated QA worktree would add duplication and constant merges. Fewer moving parts, faster for a 2-day sprint.

## 2026-07-17 · Backend worktrees reuse the main `.venv`
Backend sessions run pytest via the main worktree's interpreter rather than creating a per-worktree venv. **Why:** the venv holds multi-GB ML deps (torch/faiss/transformers); duplicating per worktree costs minutes + disk with no benefit. Code resolves from each worktree's own cwd, so isolation is preserved. Verified green from `cc2-backend`.

## 2026-07-16 · Generic skill semantics replace hardcoded lookups
`skill_semantics_service.py` classifies categories by nearest-centroid over `TECH_TAXONOMY` embeddings and estimates transfer by cosine similarity + same-category boost. **Why:** the mandate is "algorithms not examples" — the old `CATEGORY_TAXONOMY` dict and `if req in [...] and has_python` chains didn't generalize to unseen skills.

## 2026-07-16 · Two-agent runtime, LangGraph sequential
Exactly two runtime agents (Candidate Intelligence → Hiring Decision), sequential graph, no branching. **Why:** mirrors real hiring (gather evidence, then decide), keeps every decision explainable; extra agents add complexity without value.
